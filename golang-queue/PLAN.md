# golang-queue — system design

Standalone Go worker that consumes video transcode jobs from Redis, reads metadata from the shared Postgres database, transcodes via `ffmpeg` into an HLS ladder (multi-bitrate `main.m3u8`), pushes the output to S3, and writes progress back to Postgres.

This document is the architecture spec. Implementation lives in `cmd/` and `internal/` once we start coding.

---

## 1. High-level flow

```
┌──────────┐  push job   ┌─────────┐   BRPOP   ┌───────────────┐
│ Laravel  │────────────▶│  Redis  │──────────▶│  Go worker    │
└──────────┘             └─────────┘           │  (pool of N)  │
                                               └───────┬───────┘
                                                       │
                              ┌────────────────────────┼────────────────────────┐
                              │                        │                        │
                              ▼                        ▼                        ▼
                      ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
                      │   Postgres   │         │      S3      │         │    ffmpeg    │
                      │ read media/  │         │ GET source   │         │ transcode    │
                      │ profile rows │         │ PUT HLS tree │         │ HLS ladder   │
                      │ write prog.  │         └──────────────┘         └──────────────┘
                      └──────────────┘
```

### Per-job lifecycle

1. `BRPOP` a job from `queues:transcode` (blocking, with timeout for graceful shutdown).
2. Decode JSON payload. Validate required fields. If malformed → log, drop, move on (do not requeue a poison pill).
3. Open a short-lived `tx`-free Postgres read to fetch the `media_files` row and its associated `media_file_profiles` row. Verify `organization_id` matches the payload.
4. Mark `status = progress, progress = 0` in Postgres.
5. Create a per-job temp directory under `$WORK_DIR/<media_file_id>/`.
6. `HEAD` the S3 source to get size; then `GET` it (streaming) to `source.<ext>` in the temp dir. (In v1 we download to disk — ffmpeg prefers local IO, and we avoid having to presign for every read.)
7. `ffprobe` the source to get the total duration in seconds. Store it.
8. Build the ffmpeg command: one invocation, multiple `-map 0:v -map 0:a` output groups, one per rendition in the profile's `qualities` array, plus `-var_stream_map` and `-hls_segment_filename` wiring so ffmpeg produces a master `main.m3u8` plus per-rendition sub-playlists and TS (or fMP4) segments.
9. Launch ffmpeg with `-progress pipe:1 -nostats`. Scan stdout line-by-line for `out_time_us=...` and derive percent from total duration. Throttle Postgres writes (≥ 2s AND whole-percent change).
10. On ffmpeg exit success: upload the entire HLS output tree to S3 under `hls/<organization_id>/<media_file_id>/`, write `streaming_url = <s3 url to main.m3u8>`, `status = success`, `progress = 100`.
11. On error (ffmpeg non-zero, upload fail, context cancel): write `status = failed`, leave `progress` as-is, attach the tail of ffmpeg stderr to the worker log (not the DB).
12. `defer` remove the temp dir. Always. Even on panic.

## 2. Job contract (Redis)

### Queue key

- Primary key: `queues:transcode` (Redis list).
- Push with `LPUSH`, pop with `BRPOP` (FIFO from the worker's perspective).
- A second list `queues:transcode:failed` holds jobs that hit a non-retryable decode error, for manual inspection. This is written by the Go worker, not Laravel.

Why a plain list and not Redis Streams? v1 has one consumer group (the Go worker pool), jobs are not replayed, and we don't need per-message ack semantics beyond "pulled it, now it's mine". Streams add ops burden we don't need yet. Revisit if we need multi-consumer fan-out or DLQ replay.

### Payload (JSON)

```json
{
  "job_id": "01JABCXYZ...",           // ULID, generated by Laravel
  "media_file_id": "uuid",            // media_files.id
  "media_file_profile_id": "uuid",    // media_file_profiles.id — carries the qualities array
  "organization_id": "uuid",          // must match media_files.organization_id
  "enqueued_at": "2026-04-14T10:00:00Z"
}
```

Rules:
- All fields required. Missing → worker logs + drops (moves to `queues:transcode:failed`).
- The worker **does not** trust `qualities` from the payload — it always re-reads them from `media_file_profiles` so a user-driven profile change between enqueue and dequeue is honored.
- `job_id` is only for tracing; it has no DB row.

### Who pushes?

Out of scope for this doc. Laravel will dispatch after `manage.files.multipart.complete` succeeds and the user has attached a profile. This will be wired in a later PR; the Go side does not care who the producer is as long as the payload shape matches.

## 3. Data model touched

### Read

- `media_files` — id, organization_id, file_path (S3 key of source), file_name, size, status.
- `media_file_profiles` — id, media_file_id, qualities (JSON array of `VideoQuality` enum string values).
- Optionally `organizations` — only if we later need org-level S3 overrides. v1 uses one bucket.

### Write

- `media_files.status` — `progress` | `success` | `failed`.
- `media_files.progress` — 0–100 (clamped).
- `media_files.streaming_url` — final master playlist URL on S3.
- `media_files.updated_at` — touched with every write.

Nothing else. Profiles, folders, users, organizations are read-only.

## 4. Quality → ffmpeg mapping

Mirror of `App\Enums\VideoQuality` lives in `internal/quality/quality.go`. Keep it in lockstep with the PHP enum (same string keys, same width/height/bitrate).

| value   | width × height | video bitrate | audio bitrate |
|---------|----------------|---------------|---------------|
| 240p    | 352 × 240      | 600 kbps      | 64 kbps       |
| 360p    | 640 × 360      | 800 kbps      | 96 kbps       |
| 480p    | 842 × 480      | 1400 kbps     | 128 kbps      |
| 720p    | 1280 × 720     | 2800 kbps     | 128 kbps      |
| 1080p   | 1920 × 1080    | 5000 kbps     | 192 kbps      |
| 1440p   | 2560 × 1440    | 8000 kbps     | 192 kbps      |
| 2160p   | 3840 × 2160    | 25000 kbps    | 192 kbps      |

Audio bitrates above are Go-side additions — PHP enum doesn't track them. Keep them here, not in Laravel.

### ffmpeg invocation (sketch)

For a profile of `[360p, 720p, 1080p]`:

```
ffmpeg -hide_banner -y -i source.mp4 \
  -filter_complex "[0:v]split=3[v1][v2][v3]; \
                   [v1]scale=w=640:h=360[v1out]; \
                   [v2]scale=w=1280:h=720[v2out]; \
                   [v3]scale=w=1920:h=1080[v3out]" \
  -map "[v1out]" -c:v:0 libx264 -b:v:0 800k -maxrate:v:0 856k -bufsize:v:0 1200k \
  -map "[v2out]" -c:v:1 libx264 -b:v:1 2800k -maxrate:v:1 2996k -bufsize:v:1 4200k \
  -map "[v3out]" -c:v:2 libx264 -b:v:2 5000k -maxrate:v:2 5350k -bufsize:v:2 7500k \
  -map a:0 -map a:0 -map a:0 \
  -c:a aac -b:a:0 96k -b:a:1 128k -b:a:2 192k \
  -preset veryfast -g 48 -keyint_min 48 -sc_threshold 0 \
  -f hls -hls_time 6 -hls_playlist_type vod \
  -hls_segment_filename "v%v/segment_%03d.ts" \
  -master_pl_name main.m3u8 \
  -var_stream_map "v:0,a:0 v:1,a:1 v:2,a:2" \
  "v%v/playlist.m3u8" \
  -progress pipe:1 -nostats
```

Notes:
- Command is built programmatically from the profile — don't hand-roll for each quality.
- `-g 48 -keyint_min 48 -sc_threshold 0` enforces a 2-second GOP @ 24fps for clean HLS segmenting. Adjust if we ever support 60fps input.
- `-preset veryfast` is the v1 default. Tune later based on observed encode speed vs file size.
- Encoder can be swapped to `h264_videotoolbox` (mac) / `h264_nvenc` (GPU) via a `FFMPEG_VIDEO_CODEC` env var. Keep the builder codec-agnostic.

### Progress parsing

`-progress pipe:1` emits groups of `key=value\n` lines terminated by `progress=continue` or `progress=end`. We care about:

- `out_time_us` — microseconds encoded so far (across all outputs, but ffmpeg reports a single unified value).
- `progress` — `continue` or `end`.

Percent = `clamp((out_time_us / 1_000_000) / duration_seconds * 100, 0, 99)` while running, force `100` on `progress=end` + clean exit.

## 6. Concurrency & resource model

- A single process runs `WORKER_CONCURRENCY` goroutines, each in its own BRPOP loop. ffmpeg is CPU-heavy, so the default is 1; operators set it higher on bigger boxes.
- Each worker goroutine owns its temp dir, its ffmpeg process, and its DB progress writer — no shared mutable state between workers beyond the pgx pool and the Redis client.
- Graceful shutdown: on `SIGTERM`/`SIGINT`, stop BRPOP'ing new jobs, cancel the context of any in-flight ffmpeg, wait up to `SHUTDOWN_GRACE` (30s default) for clean exits, then force-kill. Any job in flight is **not** re-pushed — it will be retried at the next run only if the producer side (Laravel) has an outer retry policy. v1 accepts at-most-once semantics; we'll add an in-progress registry later if we need at-least-once.
- No distributed locking. If Laravel accidentally double-dispatches, we'll transcode twice and the second one overwrites the first. Acceptable for v1.

## 7. S3 layout

Source (uploaded via the multipart flow, unchanged):
```
<bucket>/uploads/<organization_id>/<media_file_id>/<file_name>
```

Output:
```
<bucket>/hls/<organization_id>/<media_file_id>/main.m3u8
<bucket>/hls/<organization_id>/<media_file_id>/v0/playlist.m3u8
<bucket>/hls/<organization_id>/<media_file_id>/v0/segment_000.ts
<bucket>/hls/<organization_id>/<media_file_id>/v1/playlist.m3u8
...
```

`media_files.streaming_url` stores the fully-qualified URL to `main.m3u8` (resolved via the same `AWS_URL` the Laravel app uses, if set).

Uploads use `PutObject` per file, in parallel, bounded to ~8 concurrent. Segments are small enough that multipart upload for each is overkill.

## 8. Failure handling

| failure | action |
|---|---|
| malformed payload | log, push to `queues:transcode:failed`, do **not** touch any DB row |
| `media_files` row missing / org mismatch | log, drop, no DB write |
| source download fails | `status = failed`, log error, move on |
| ffprobe fails | `status = failed`, log, move on |
| ffmpeg exits non-zero | `status = failed`, log stderr tail, move on |
| S3 upload partial failure | retry each failed `PutObject` up to 3× with backoff; if still failing, `status = failed` |
| DB unreachable during progress write | log + drop that progress update; continue encoding; final success write will retry |
| DB unreachable during terminal write | retry 5× with backoff, then log error and exit the worker (let the supervisor restart) |

No in-Go retry queue. Re-running the same job is the operator's responsibility (re-dispatch from Laravel). v2 can add a retry count + exponential backoff sidecar.

## 9. Observability

- Structured logs via `log/slog` with JSON handler. Fields: `job_id`, `media_file_id`, `org_id`, `stage`, `duration_ms`.
- `stage` taxonomy: `dequeue`, `load`, `download`, `probe`, `transcode`, `upload`, `finalize`, `cleanup`.
- No metrics server in v1. If we need Prometheus later, wire a `/metrics` http handler on a separate port — don't add it preemptively.
- ffmpeg stderr is captured to a ring buffer (last 200 lines) and only emitted on failure.

---

## 10. Out of scope (v1)

- Thumbnail / sprite generation.
- DRM / encrypted HLS.
- Live streaming.
- Resumable / chunked encoding.
- Multi-region S3.
- Webhook notifications on completion (Laravel will poll `media_files.status`).
- Any kind of admin UI for the worker itself.

These are all reasonable v2 topics. Don't build them now — the goal of v1 is "user uploads a video, within N minutes `status = success` and `streaming_url` plays in an HLS player".

## 11. Open questions (resolve before coding)

1. **Who dispatches the job?** Probably `ManageController::complete` after it writes the `media_files` row, gated on the user having selected a profile. Confirm with the user before wiring.
2. **Which profile?** The media file's `media_file_profiles` row — but how is it populated today? If the user hasn't attached a profile, do we fall back to the org default `profiles` row? Need to walk the existing Laravel flow.
3. **Source URL field.** `media_files.file_path` stores the S3 key; `source_url` is for URL-based ingests. Worker should prefer `file_path`, and only fall back to `source_url` if `file_path` is null. Confirm this matches reality.
4. **Bucket sharing.** The worker assumes the same bucket as Laravel. If outputs should live in a separate bucket (e.g. a CDN-fronted one), add `AWS_OUTPUT_BUCKET`.
5. **Hardware acceleration.** Are we deploying on machines with GPUs / VideoToolbox? This dictates the default `FFMPEG_VIDEO_CODEC`.

Answer these in this document (edit the section, don't leave it open) before the first line of Go is written.
